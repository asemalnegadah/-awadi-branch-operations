## 1. الحكم التنفيذي

المستودع ليس مشروعًا فارغًا، ولا يجوز إعادة تأسيسه من الصفر. توجد نواة قوية نسبيًا لسلامة البيانات المالية، هوية العميل، التحصيل، عهدة المندوب، رفع ملفات PDF، استخراج تقارير Onyx، وحفظ نتائج الاستخراج في PostgreSQL.

في المقابل، النظام ليس تطبيقًا تشغيليًا مكتملًا للاستخدام اليومي بعد. معظم الموجود حاليًا هو:

- مخطط PostgreSQL وقيود سلامة.
- Domain logic قابل للاختبار.
- Repositories جزئية.
- اختبارات Unit وPostgreSQL وIntegration.
- صفحة تأسيسية واحدة وواجهة Health واحدة.

أكبر عوائق الاستخدام الفعلي هي غياب المصادقة والجلسات والصلاحيات المطبقة على الخادم، غياب APIs وواجهات التشغيل، وعدم اكتمال طبقات التطبيق والتخزين لمعظم دورات العمل.

## 2. قواعد لا يجوز كسرها

1. النظام لفرع عدن فقط؛ لا يضاف `branch_id` أو `tenant_id` في هذه المرحلة.
2. تبقى SR وRG منفصلتين في الحسابات والحركات والتقارير.
3. تخزن الأموال بوحدات صغرى صحيحة، ولا تستخدم أعداد JavaScript العشرية للحركات المالية.
4. دفتر الحركات Append-only؛ التصحيح بالحركة العكسية وليس بالحذف أو تعديل القيد المعتمد.
5. لا تعدل migrations من `0001` إلى `0011`؛ أي تغيير جديد يبدأ بالرقم التالي المتاح.
6. لا تدخل بيانات تشغيلية حقيقية أو أسرار أو مفاتيح API في المستودع.
7. لا تعاد كتابة وظيفة موجودة باسم آخر؛ تستكمل طبقاتها الناقصة.
8. النظام يبدأ بوضع مستخدم واحد هو مدير الفرع، مع بقاء جميع انتقالات الحالة وسجل التدقيق.
9. إضافة المستخدمين لاحقًا يجب ألا تتطلب إعادة بناء المخطط أو منطق الوحدات.
10. الذكاء الاصطناعي يقترح ويشرح فقط، ولا يعتمد خطة ولا يرحل حركة مالية.

قبل هذا الفرع كان CI يستخدم `npm install` ولا يوجد `package-lock.json`، ولذلك لم يكن تثبيت شجرة الاعتماديات قابلًا لإعادة الإنتاج بالكامل. كما لم يوجد فحص آلي يمنع:

- تكرار رقم migration.
- وجود فجوة في التسلسل.
- تسمية migration بطريقة غير قياسية.
- ربط اختبار قاعدة بيانات برقم migration غير موجود.

## 6. الفجوات الحرجة قبل البيانات الحقيقية

### P0.1 المصادقة والجلسات

- لا Login أو Logout.
- لا جدول جلسات أو إبطال جلسات.
- لا خدمة كلمات مرور.
- لا قفل لمحاولات الدخول.
- لا حماية للمسارات التشغيلية.

### P0.2 الصلاحيات المطبقة

- ثوابت الأدوار والصلاحيات موجودة، لكن `role_permissions` غير معبأة افتراضيًا بصورة مكتملة.
- لا Authorization layer موحدة في Route Handlers والخدمات.
- لا Data scope فعلي يمنع المندوب من الوصول إلى عملاء مندوب آخر.
- لا يجوز الوثوق بصلاحيات يرسلها العميل.

### P0.3 وضع المستخدم الواحد

الوضع المعتمد حاليًا هو `SINGLE_MANAGER`:

- مدير الفرع ينشئ ويراجع ويعتمد ويغلق بنفسه.
- لا تحذف مراحل الحالة.
- يسجل كل انتقال منفصلًا.
- تسجل العمليات ذات الاعتماد الذاتي بوضوح.
- لا تخفف Transactions أو idempotency أو قواعد العملات أو Audit.
- عند إضافة المستخدمين مستقبلًا يمكن تفعيل الفصل بين الأدوار دون إعادة بناء.

### P0.4 العمليات المالية الذرية

- عكس التحصيل يجب أن يعكس الدفتر والتخصيص والعهدة وأثر الصندوق في transaction واحدة.
- يجب منع العكس المكرر.
- إنشاء المستند المالي وترحيله يجب ألا يترك حالة جزئية عند فشل خطوة واحدة.

### P0.5 الاستيراد

- يجب ألا تعتمد حالة الملف منفصلة عن حالة صفوفه.
- يجب قفل بيانات الصف بعد الاعتماد أو إنشاء إصدار تصحيحي.
- ترقية snapshot إلى CURRENT وخفض السابق إلى HISTORICAL يجب أن تكون ذرية مع قفل للسلسلة.

## 9. مخاطر الأمن والتشغيل الحالية

- لا توجد مصادقة أو جلسات فعلية.
- Security headers جزئية؛ لا توجد CSP أو HSTS إنتاجية موثقة.
- لا Rate limiting أو CSRF protection لمسارات الكتابة.
- لا Storage خاص أو Signed URLs.
- فحص PDF الحالي heuristic وليس Malware scanner.
- لا structured logging أو error monitoring.
- Health لا يتحقق من migrations أو التخزين أو الـWorker.
- لا Dockerfile أو runbook نشر واستعادة مكتمل.
- لا اختبار استعادة نسخ احتياطية.

## 11. ما نفذ في الدفعة صفر

- إنشاء فرع مستقل من أحدث `main`.
- إنشاء `package-lock.json` فعلي بواسطة Node 22 وnpm.
- تحويل CI إلى `npm ci`.
- جعل npm cache يعتمد على `package-lock.json`.
- إضافة `scripts/verify-migrations.mjs`.
- إدخال فحص تسلسل migrations ضمن `npm run check` وCI.
- الحفاظ على PostgreSQL 16 واختبارات قاعدة البيانات والتكامل والبناء.
- إنشاء Pull Request مسودة وعدم الدمج إلى `main`.

## 12. معيار اكتمال الدفعة صفر

لا تعتبر الدفعة صفر جاهزة إلا بعد:

1. نجاح `npm ci` في GitHub Actions.
2. نجاح فحص migration sequence.
3. نجاح lint.
4. نجاح typecheck.
5. نجاح Unit tests.
6. تطبيق جميع migrations على PostgreSQL 16 نظيفة.
7. نجاح جميع PostgreSQL integrity tests.
8. نجاح Integration tests.
9. نجاح Next.js build.
10. مراجعة diff والتأكد من عدم وجود أسرار أو بيانات حقيقية.

حتى تحقق هذه الشروط يبقى Pull Request مسودة وغير صالح للدمج.
