# مراجعة شاملة للنواة قبل ربط رفع PDF

التاريخ: 2026-07-14

## النطاق الذي تمت مراجعته

- بنية التطبيق وCI.
- فصل SR وRG وكائنات المبالغ.
- المستخدمون والأدوار والصلاحيات وسجل التدقيق.
- العملاء والهوية والتكرار وتعيين المندوب.
- دفتر العميل المالي والحركات العكسية.
- التحصيلات والتخصيصات وعهدة المندوب.
- اتصال PostgreSQL ومستودع مسودات التحصيل.
- تسجيل الملفات ومنطقة الاستخراج والمراجعة.
- مطابقة رقم العميل والأسماء المقطوعة.
- النسخ الزمنية لكشوف أعمار الديون.
- محلل إحداثيات PDF الخاص بتقرير Onyx.

## النتيجة

الأساس المعماري صحيح ومتوافق مع فرع عدن الواحد، وفصل العملات، وعدم حذف الحركات المالية، والمراجعة قبل ترحيل نتائج الملفات. اجتازت المراحل السابقة Lint وTypecheck والاختبارات وPostgreSQL وBuild قبل دمجها.

لكن المراجعة كشفت نقاط تقوية يجب إغلاقها بالتزامن مع الخطوة التالية:

1. عميل PostgreSQL يجب أن يعاد استخدامه في الإنتاج لتجنب إنشاء pool جديد مع كل استدعاء.
2. التحقق من تواريخ ISO في مقارنة نسخ التقارير يجب أن يرفض تواريخ مثل 2026-02-31 وأن يرفض بداية بعد النهاية.
3. البيانات الزمنية للملف وحقول هوية snapshot يجب ألا يعاد كتابتها بعد تثبيتها.
4. إنشاء التحصيل مباشرة في قاعدة البيانات يجب أن يبدأ دائمًا من DRAFT، حتى لا يمكن تجاوز دورة الحالات بإدخال مباشر.
5. إنشاء الملف مباشرة يجب أن يبدأ دائمًا من REGISTERED.
6. يجب منع أكثر من تعيين دور فعال واحد للمستخدم والدور نفسيهما.
7. حالات مراجعة صفوف الاستخراج يجب أن تخضع لقيود تمنع اعتماد صف غير مراجع أو تغيير المصدر الخام.

## القرار

لا تمنع هذه النقاط استمرار البناء، لكنها تعامل كتصحيحات إلزامية قبل توصيل واجهة رفع عامة أو بيانات تشغيلية حقيقية. ستنفذ داخل مرحلة رفع ومعالجة PDF مع اختبارات PostgreSQL ووحدات قبل الدمج.
