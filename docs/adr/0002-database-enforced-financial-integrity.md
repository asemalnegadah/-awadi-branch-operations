# ADR-0002: فرض سلامة العمليات المالية داخل PostgreSQL

- **الحالة:** معتمد
- **التاريخ:** 2026-07-14
- **النطاق:** التحصيلات، التخصيصات، دفتر العميل، عهدة المندوب وسجل الحالات

## القرار

لا تعتمد سلامة العمليات المالية على واجهة المستخدم أو TypeScript وحدهما. يجب أن تفرض قاعدة PostgreSQL القيود الجوهرية التي تمنع الحذف، خلط العملات، القفز بين الحالات، تجاوز مبالغ التخصيص، ربط حركة دفتر غير مطابقة، وتسليم نقدية تتجاوز رصيد العهدة.

## المبررات

- يمكن أن تحتوي أي طبقة تطبيقية على خطأ.
- يمكن أن تعمل عدة طلبات متزامنة على السجل نفسه.
- واجهات الاستيراد والمهام الخلفية قد تتجاوز شاشة المستخدم.
- قاعدة البيانات هي آخر حاجز قبل فساد السجل المالي.

## النتائج الملزمة

1. جميع Migrations تطبق آليًا على PostgreSQL حقيقية داخل CI.
2. تجرى اختبارات سلامة SQL بعد تطبيق Migrations.
3. التحصيل لا يحذف، وتعديل بياناته المالية مسموح في `DRAFT` فقط.
4. انتقال الحالة يسجل آليًا في سجل غير قابل للتعديل.
5. المراجع والمعتمد ومستلم الأموال يخضعون لقواعد الفصل داخل قاعدة البيانات.
6. حركة الدفتر المرتبطة بالتحصيل يجب أن تطابق العميل والحساب والعملة والمبلغ والاتجاه والمصدر.
7. تخصيصات التحصيل لا تتجاوز مبلغ التحصيل، ولا تغير عملته.
8. رصيد عهدة المندوب يحسب ويقفل داخل المعاملة قبل حركة الخروج.
9. الحركات العكسية تستخدم بدل الحذف أو التعديل التاريخي.

## أثر القرار

يزداد تعقيد Migrations والاختبارات، لكنه يقلل خطر فساد البيانات ويجعل النظام أكثر تحملًا لأخطاء التطبيق والتزامن والاستيراد.
