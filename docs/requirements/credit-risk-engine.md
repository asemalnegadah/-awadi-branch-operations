# متطلبات محرك المخاطر والمنع الائتماني

## 1. النطاق المعماري

- النظام مخصص لفرع عدن فقط.
- لا يضاف `branch_id` أو `tenant_id` أو أي منطق متعدد الفروع.
- كل تقييم وقرار واستثناء واستهلاك مستقل حسب حساب العميل والعملة.
- لا تجمع SR مع RG في درجة أو رصيد أو حد أو قرار واحد.
- لا يغير محرك المخاطر دفتر الحركات أو رصيد العميل.
- الذكاء الاصطناعي قد يشرح النتيجة لاحقًا، لكنه لا يفعّل منعًا ولا يعتمد استثناءً.

## 2. مصادر الخطر

يدعم الإصدار `credit-risk-v1` عوامل قابلة للتفسير تشمل:

- أعمار الدين 31–60، 61–90، 91–180، وأكثر من 180 يومًا.
- الوعود المكسورة أو المتأخرة.
- تجاوز الحد الائتماني.
- توقف العميل أو إغلاقه أو إفلاسه أو وجود نزاع.
- المطابقات غير المحسومة.
- نقص وسيلة اتصال صالحة.
- طول المدة منذ آخر زيارة.
- تحصيلات لم يكتمل تسليمها أو إغلاق عهدتها.

المصادر المتاحة تُشتق من دفتر الحركات والوعود والتحصيلات وملف العميل على الخادم. العوامل التي لم تتوفر بياناتها تظهر في `missing_inputs` ولا تستبدل بافتراضات مخفية.

## 3. تقييم المخاطر

كل تقييم Snapshot غير قابل للتعديل ويتضمن:

- العميل وحسابه والعملة.
- وقت قطع البيانات ووقت الحساب من الخادم.
- إصدار قواعد الحساب.
- درجة من 0 إلى 100.
- مستوى `LOW | MEDIUM | HIGH | CRITICAL`.
- توصية `NONE | MONITOR | LIMIT | BLOCK`.
- قائمة العوامل والنقاط والقيم المرصودة والشرح العربي.
- درجة جودة البيانات والمدخلات الناقصة.
- بصمة المدخلات وSnapshot المصادر.
- المستخدم ومعرف الطلب ومفتاح منع التكرار.

الحدود:

- 0–24: منخفض، دون إجراء.
- 25–49: متوسط، متابعة.
- 50–74: مرتفع، تقييد ائتماني مقترح.
- 75–100: حرج، منع آجل مقترح.

التوصية لا تصبح قرارًا نافذًا تلقائيًا. إعادة الحساب تقفل حساب العميل وتُسلسل التقييمات المتزامنة، ويعيد `idempotency_key` نفسه النتيجة الأصلية.

## 4. قرارات المنع أو التقييد

الأنواع:

- `LIMIT`: سقف للرصيد المتوقع بعد إضافة العملية الجديدة، وليس حدًا لكل فاتورة منفردة.
- `SUSPEND`: إيقاف مؤقت للبيع الآجل.
- `BLOCK`: منع كامل للبيع الآجل.

المسار:

`DRAFT → PENDING_APPROVAL → ACTIVE → REVOKED | EXPIRED`

ومسار الرفض:

`PENDING_APPROVAL → REJECTED`

القواعد:

- لا يوجد أكثر من قرار نافذ لحساب وعملة في الوقت نفسه.
- القرار يحتفظ بسبب مرمز وشرح ومصدر ووقت مراجعة وشروط إعادة السماح.
- لا يعتمد المقترح قراره بنفسه في وضع `MULTI_USER`.
- يسمح اعتماد المدير نفسه فقط في `SINGLE_MANAGER` وفق الدور الفعلي المسجل.
- المرسل والمعتمد ووقتاهما يصبحون غير قابلين للتغيير بعد تسجيلهم.
- لا حذف للقرار؛ الإلغاء أو الانتهاء انتقال موثق.
- لا ينتهي القرار الأب قبل إلغاء أو انتهاء استثناءاته النافذة.
- جميع الأحداث Append-only.

## 5. الاستثناءات والاستهلاك

الاستثناء مرتبط بقرار نافذ ويحدد:

- الحساب والعملة.
- المبلغ الأقصى بوحدات صغرى.
- المدة من/إلى.
- النطاق: عملية واحدة أو عدة عمليات.
- السبب والشروط.
- المقترح والمعتمد وحالة الاستثناء.

المسار:

`DRAFT → PENDING_APPROVAL → ACTIVE → REVOKED | EXPIRED`

أو:

`PENDING_APPROVAL → REJECTED`

لا يتجاوز الاستثناء مدة القرار المرتبط، ولا يغير القرار الأصلي أو التقييم.

كل استخدام يسجل في `credit_exception_usage_entries` كدفتر Append-only يتضمن المصدر والمبلغ والعملة والمنفذ والطلب ومفتاح منع التكرار. التصحيح يتم بحركة `REVERSE` مساوية ومقابلة، ولا تعدل حركة الاستخدام الأصلية.

قواعد الاستخدام:

- `SINGLE_TRANSACTION`: يسمح باستهلاك فعلي واحد فقط ما لم يُعكس الاستهلاك الأصلي.
- `MULTIPLE_TRANSACTIONS`: لا يتجاوز صافي الاستخدام التراكمي المبلغ المعتمد.
- قفل صف الاستثناء يمنع استهلاكين متزامنين من تجاوز الحد.
- لا يعاد استخدام مصدر بيع واحد مرتين.
- فحص السماح بالبيع يأخذ الاستهلاك السابق والرصيد المتوقع في الاعتبار.

## 6. الصلاحيات

- `risk.read`
- `risk.recalculate`
- `risk.view_history`
- `credit_restrictions.propose`
- `credit_restrictions.approve`
- `credit_restrictions.revoke`
- `credit_exceptions.propose`
- `credit_exceptions.approve`
- `credit_exceptions.revoke`
- `credit_exceptions.consume`

`SYSTEM_ADMIN` لا يحصل تلقائيًا على بيانات المخاطر أو الديون أو صلاحيات القرار المالي.

## 7. الطبقات التشغيلية

الوحدة تشمل:

- محرك نقاط حتمي واختبارات وحدات.
- Migrations `0023` و`0024` و`0025`.
- PostgreSQL repositories للتقييم والقرارات والاستثناءات والاستهلاك.
- خدمة صلاحيات ونطاق مندوب.
- APIs للقراءة وإعادة الحساب وفحص البيع الآجل ودورة القرارات والاستثناءات والاستهلاك والعكس.
- واجهة عربية RTL للقائمة وتفاصيل الحساب والعوامل والتاريخ والإجراءات.
- ربط الوحدة بلوحة مدير الفرع.
- سجل تدقيق وأحداث ودفاتر Append-only.

## 8. معايير القبول

- لا توجد أعمدة فرع أو مستأجر.
- لا يمكن تعديل أو حذف تقييم مخاطر أو حدث أو حركة استهلاك بعد إنشائها.
- لا يمكن إنشاء تقييم بعملة لا تطابق حساب العميل.
- لا يمكن إنشاء تقييمين متزامنين يكسران تسلسل `supersedes_assessment_id`.
- لا يمكن تفعيل قرارين للحساب والعملة نفسيهما.
- لا يمكن الانتقال مباشرة من مسودة إلى قرار نافذ دون تقديم واعتماد.
- لا يمكن الاعتماد الذاتي في `MULTI_USER`.
- لا يمكن تغيير هوية المرسل أو المعتمد بعد تسجيلها.
- الاستثناء لا يتجاوز مبلغًا أو مدة محددين ولا يرتبط بقرار غير نافذ.
- لا يمكن إعادة استخدام استثناء عملية واحدة بعد استهلاكه.
- قرار `LIMIT` يفحص الرصيد القائم مضافًا إليه مبلغ العملية الجديدة.
- جميع المبالغ أعداد صحيحة بوحدات صغرى.
- تعمل صلاحيات السجل للمندوب عبر تكليفه النشط.
- تمر Unit وPostgreSQL وIntegration وNext.js وCloudflare وCodeQL وSecret Scan قبل الدمج.
