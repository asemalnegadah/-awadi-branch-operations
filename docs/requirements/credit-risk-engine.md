# متطلبات محرك المخاطر والمنع الائتماني

## 1. النطاق المعماري

- النظام مخصص لفرع عدن فقط.
- لا يضاف `branch_id` أو `tenant_id` أو أي منطق متعدد الفروع.
- كل تقييم وقرار واستثناء مستقل حسب حساب العميل والعملة.
- لا تجمع SR مع RG في درجة أو رصيد أو حد أو قرار واحد.
- لا يغير محرك المخاطر دفتر الحركات أو رصيد العميل.
- الذكاء الاصطناعي قد يشرح النتيجة لاحقًا، لكنه لا يفعّل منعًا ولا يعتمد استثناءً.

## 2. مصادر الخطر

يدعم الإصدار الأول عوامل قابلة للتفسير تشمل:

- أعمار الدين 31–60، 61–90، 91–180، وأكثر من 180 يومًا.
- الوعود المكسورة أو المتأخرة.
- تجاوز الحد الائتماني.
- توقف العميل أو إغلاقه أو إفلاسه أو وجود نزاع.
- المطابقات غير المحسومة.
- نقص وسيلة اتصال صالحة.
- طول المدة منذ آخر زيارة.
- تحصيلات لم يكتمل تسليمها أو إغلاق عهدتها.

تظل العوامل التي لم تتوفر بياناتها ظاهرة في `missing_inputs` ولا تُستبدل بافتراضات مخفية.

## 3. تقييم المخاطر

كل تقييم هو Snapshot غير قابل للتعديل ويتضمن:

- العميل وحسابه والعملة.
- وقت قطع البيانات ووقت الحساب من الخادم.
- إصدار قواعد الحساب.
- درجة من 0 إلى 100.
- مستوى `LOW | MEDIUM | HIGH | CRITICAL`.
- توصية `NONE | MONITOR | LIMIT | BLOCK`.
- قائمة العوامل والنقاط والقيم المرصودة والشرح العربي.
- درجة جودة البيانات والمدخلات الناقصة.
- بصمة المدخلات ومراجع المصادر.
- المستخدم والطلب ومفتاح منع التكرار.

حدود الإصدار `credit-risk-v1`:

- 0–24: منخفض، دون إجراء.
- 25–49: متوسط، متابعة.
- 50–74: مرتفع، تقييد ائتماني مقترح.
- 75–100: حرج، منع آجل مقترح.

التوصية لا تصبح قرارًا نافذًا تلقائيًا.

## 4. قرارات المنع أو التقييد

الأنواع:

- `LIMIT`: حد آجل محدد بوحدات صغرى ولعملة واحدة.
- `SUSPEND`: إيقاف مؤقت للبيع الآجل.
- `BLOCK`: منع كامل للبيع الآجل.

المسار:

`DRAFT → PENDING_APPROVAL → ACTIVE → REVOKED | EXPIRED`

ومسار الرفض:

`PENDING_APPROVAL → REJECTED`

القواعد:

- لا يوجد أكثر من قرار نافذ لحساب وعمـلة في الوقت نفسه.
- القرار يحتفظ بسبب مرمز وشرح ومصدر ووقت مراجعة وشروط إعادة السماح.
- لا يعتمد المقترح قراره بنفسه في وضع `MULTI_USER`.
- يسمح استثناء المستخدم الواحد فقط وفق قاعدة `SINGLE_MANAGER` المسجلة والمدققة.
- لا حذف للقرار؛ الإلغاء أو الانتهاء انتقال موثق.
- جميع الأحداث Append-only.

## 5. الاستثناءات

الاستثناء مرتبط بقرار نافذ ويحدد:

- الحساب والعملة.
- المبلغ الأقصى بوحدات صغرى.
- المدة من/إلى.
- النطاق: عملية واحدة أو عدة عمليات.
- السبب والشروط.
- المقترح والمعتمد وحالة الاستثناء.

المسار:

`DRAFT → PENDING_APPROVAL → ACTIVE → REVOKED | EXPIRED`

أو:

`PENDING_APPROVAL → REJECTED`

لا يتجاوز الاستثناء مدة القرار المرتبط، ولا يغير القرار الأصلي أو التقييم.

## 6. الصلاحيات

- `risk.read`
- `risk.recalculate`
- `risk.view_history`
- `credit_restrictions.propose`
- `credit_restrictions.approve`
- `credit_restrictions.revoke`
- `credit_exceptions.propose`
- `credit_exceptions.approve`
- `credit_exceptions.revoke`

`SYSTEM_ADMIN` لا يحصل تلقائيًا على بيانات المخاطر أو الديون.

## 7. مراحل التنفيذ

### الدفعة الأولى

- أنواع Domain ومحرك نقاط حتمي واختبارات Unit.
- جداول التقييم والقرارات والاستثناءات والأحداث.
- قيود الهوية والعملة والحالات والفصل بين المقترح والمعتمد.
- اختبارات PostgreSQL للسلوك والـAppend-only.

### الدفعة الثانية

- PostgreSQL repository وخدمة صلاحيات ونطاق المندوب.
- تجميع المدخلات من دفتر الحركات والوعود والمطابقات والعهدة.
- APIs للقراءة وإعادة الحساب واقتراح/اعتماد/إلغاء القرارات والاستثناءات.

### الدفعة الثالثة

- واجهات عربية RTL للوحة المخاطر وCustomer 360 وسجل القرارات.
- حارس موحد `assertCreditSaleAllowed` يُربط بإنشاء الفاتورة الآجلة عند اكتمال وحدة المبيعات.
- تقارير العملاء الخطرين والممنوعين والاستثناءات القريبة من الانتهاء.

## 8. معايير القبول للدفعة الأولى

- لا توجد أعمدة فرع أو مستأجر.
- لا يمكن تعديل أو حذف تقييم مخاطر أو حدث بعد إنشائه.
- لا يمكن إنشاء تقييم بعملة لا تطابق حساب العميل.
- لا يمكن تفعيل قرارين للحساب والعملة نفسيهما.
- لا يمكن الانتقال مباشرة من مسودة إلى قرار نافذ دون تقديم واعتماد.
- لا يمكن الاعتماد الذاتي في `MULTI_USER`.
- الاستثناء لا يتجاوز مبلغًا أو مدة محددين ولا يرتبط بقرار غير نافذ.
- جميع المبالغ أعداد صحيحة بوحدات صغرى.
- اختبارات Unit وPostgreSQL وCI تمر قبل طلب المراجعة.
