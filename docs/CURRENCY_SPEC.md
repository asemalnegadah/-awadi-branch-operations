# مواصفة العملات والحسابات

**الحالة:** مسودة تأسيسية ملزمة تقنيًا  
**النطاق:** جميع الحركات المالية والتقارير

## 1. العملات التشغيلية

يدعم النظام حسابين ماليين منفصلين فقط في الإصدار الأول:

- `SR`
- `RG`

الأسماء التجارية أو المحاسبية الكاملة لكل رمز تعتمد لاحقًا من الإدارة والحسابات، لكن الرموز الداخلية ثابتة ولا تستخدم كنص حر.

## 2. نموذج التخزين

كل مبلغ يخزن كالتالي:

- `currency_code`: قيمة ثابتة `SR` أو `RG`.
- `amount_minor`: عدد صحيح يمثل الوحدات الصغرى.

يمنع تخزين الأموال باستخدام الأعداد العشرية العائمة في JavaScript.

يعتمد مبدئيًا منزلتان عشريتان لكل عملة إلى أن تعتمد الحسابات سياسة مختلفة. أي تغيير لاحق يحتاج Migration واختبارات ترحيل.

## 3. قواعد الفصل

- لا تحتوي حركة مالية واحدة على أكثر من عملة.
- لا تحتوي فاتورة واحدة على أكثر من عملة.
- لا يحتوي سند قبض واحد على أكثر من عملة.
- لا تجمع التقارير العملتين في إجمالي مالي واحد.
- يمكن عرض مؤشرين متجاورين، أحدهما SR والآخر RG.
- للعميل دفتر مستقل منطقيًا لكل عملة.
- للصندوق والعهدة والإغلاق أرصدة منفصلة لكل عملة.

## 4. العمليات الحسابية

- الجمع والطرح مسموحان فقط عند تطابق العملة.
- أي محاولة جمع أو طرح عملتين مختلفتين تفشل قبل الكتابة إلى قاعدة البيانات.
- التقريب يتم وفق عدد المنازل المعتمد للعملة.
- لا يسمح بمبالغ `NaN` أو `Infinity` أو قيم خارج نطاق العدد الصحيح الآمن في طبقة التطبيق.
- قاعدة البيانات تستخدم نوعًا صحيحًا واسعًا مناسبًا للوحدات الصغرى.

## 5. التحويل بين العملات

لا يوجد تحويل تلقائي بين SR وRG في MVP.

أي حاجة مستقبلية للتحويل تتطلب:

- وثيقة سياسة صرف مستقلة.
- سعر صرف وتاريخه ومصدره.
- حسابات وسيطة وحركات مزدوجة واضحة.
- اعتمادًا من الحسابات والمدير.
- سجل تدقيق كامل.

حتى اعتماد ذلك، يمنع النظام التحويل بين الحسابين.

## 6. الترقيم والتقارير

يمكن اعتماد سلاسل مستندات منفصلة حسب العملة عند توثيق سياسة الترقيم.

يظهر في كل مستند مالي:

- رمز العملة.
- اسمها المعتمد.
- المبلغ رقمًا وكتابة عند الحاجة.
- وقت قطع البيانات.
- حالة المستند.

## 7. قيود قاعدة البيانات المطلوبة

- `currency_code` غير قابل للإهمال في كل حركة مالية.
- قيد CHECK يقبل `SR` أو `RG` فقط.
- قيود فريدة للسندات تراعي العملة وسلسلة الترقيم.
- لا تعتمد الواجهة وحدها لمنع خلط العملات.

## 8. اختبارات القبول

- يفشل جمع Money من SR مع Money من RG.
- لا تقبل API حركة بلا عملة.
- لا تقبل API رمز عملة غير معرف.
- لا تنشأ فاتورة أو سند متعدد العملات.
- لا تعرض لوحة التحكم إجماليًا يجمع العملتين.
- يحافظ الاستيراد على فصل العملات، ويحجر الصف المختلط أو غير المحدد.
