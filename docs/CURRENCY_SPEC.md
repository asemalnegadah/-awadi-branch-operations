# مواصفة العملات والحسابات

**الحالة:** مسودة تنفيذية أولى  
**النطاق:** فرع العوادي – عدن فقط

## 1. الهدف

منع خلط العملات وضمان أن كل حركة ورصيد وتقرير قابل للتدقيق والتتبع بصورة مستقلة.

## 2. العملات المعتمدة

يدعم النظام حسابين ماليين مستقلين:

- `SR`
- `RG`

الأسماء التجارية أو المحاسبية الكاملة للعملتين، وعدد المنازل العشرية، ورموز الطباعة، تعتمد لاحقًا من الإدارة والحسابات قبل تشغيل البيانات الحقيقية.

## 3. القواعد الإلزامية

1. كل حركة مالية تحمل `currency_id` واحدًا فقط.
2. لا يسمح بحركة أو فاتورة أو سند يجمع SR وRG.
3. يمكن أن يملك العميل حساب SR وحساب RG مستقلين.
4. لا تجمع لوحة التحكم أو التقارير العملتين في إجمالي مالي واحد.
5. لا يوجد تحويل تلقائي بين العملتين.
6. أي تحويل استثنائي يحتاج حركتين مستقلتين، سعرًا معتمدًا، مرجع قرار وسجل تدقيق.
7. التقريب يتم داخل وحدة المال المركزية، لا داخل الشاشات.
8. تخزن المبالغ بوحدة صحيحة صغرى `minor_units` متى تم اعتماد المنازل العشرية، لمنع أخطاء الكسور العائمة.
9. لا تستخدم أرقام JavaScript العشرية لإجراء العمليات المحاسبية النهائية.
10. رقم السند والفاتورة يمكن أن يملك سلسلة مستقلة حسب العملة إذا اعتمدت الحسابات ذلك.

## 4. نموذج العملة المقترح

`currencies`

- `id`
- `code` — فريد وثابت
- `name_ar`
- `symbol_ar`
- `decimal_places`
- `rounding_mode`
- `is_active`
- `created_at`
- `updated_at`

## 5. حساب العميل حسب العملة

`customer_accounts`

- `id`
- `customer_id`
- `currency_id`
- `status`
- `credit_limit_minor`
- `credit_status`
- `opened_at`
- `closed_at`

قيد فريد:

`UNIQUE(customer_id, currency_id)`

## 6. العرض والطباعة

كل مبلغ يعرض مع:

- العملة بوضوح.
- تنسيق ثابت.
- اتجاه عربي صحيح.
- عدم حذف الأصفار أو التقريب بصمت.
- عرض وقت قطع البيانات في التقارير الملخصة.

## 7. معايير القبول

- يرفض API أي حركة بلا عملة.
- يرفض API أي مجموعة بنود مالية بعملات مختلفة داخل مستند واحد.
- لا يمكن تخصيص تحصيل RG لفاتورة SR والعكس.
- تقارير SR لا تحتوي مبالغ RG.
- تغيير تعريف عملة لا يغير الحركات التاريخية بصمت.
