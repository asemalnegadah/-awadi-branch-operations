# المواصفة التنفيذية المرجعية لنظام تشغيل ورقابة فرع العوادي – عدن

**الإصدار:** 1.0.0  
**الحالة:** مرجع ملزم للبناء  
**النطاق:** فرع عدن فقط  
**لغة الواجهة:** العربية RTL  
**المستودع:** `asemalnegadah/-awadi-branch-operations`

---

## 1. سلطة هذه الوثيقة

هذه الوثيقة هي المرجع الرسمي الذي تُبنى عليه قاعدة البيانات، الواجهات، الـAPI، المهام الخلفية، الصلاحيات، الاستيراد، التقارير، الاختبارات، النشر، النسخ الاحتياطي والتشغيل الفعلي.

عند وجود تعارض بين كود أو شاشة أو تقرير وبين هذه الوثيقة، تُعد هذه الوثيقة هي المرجع الأعلى إلى أن يصدر تعديل موثق ومعتمد بإصدار جديد.

لا يجوز اعتبار أي وحدة مكتملة لمجرد أن واجهتها تعمل؛ يجب أن تجتاز قواعد البيانات، حالات العمليات، الصلاحيات، سجل التدقيق، الاختبارات ومعايير القبول المحددة هنا.

---

## 2. القرار المعماري الثابت: فرع عدن فقط

النظام مخصص لتشغيل ورقابة **فرع العوادي – عدن فقط**.

لا يدعم النظام:

- تعدد الفروع.
- تعدد المستأجرين Multi-Tenancy.
- مقارنة الفروع.
- نقل المخزون أو الصندوق بين فروع.
- صلاحيات حسب الفرع.
- `branch_id` أو `tenant_id` أو `company_id` داخل السجلات التشغيلية.

يسمح النظام بتعدد العناصر الداخلية داخل فرع عدن، مثل:

- المندوبين.
- المناطق.
- خطوط السير.
- المركبات.
- المخازن الداخلية.
- الصناديق.
- المستخدمين.
- فرق العمل.

يحتوي النظام على سجل إعدادات منشأة واحد باسم مقترح:

`organization_settings`

ويشمل اسم المنشأة، اسم الفرع، العنوان، الشعار، الترويسة، الهواتف، المنطقة الزمنية، العملات وإعدادات التقارير.

---

## 3. هوية النظام ودورة العمل

الاسم التشغيلي:

**نظام تشغيل ورقابة فرع العوادي – عدن**

النظام ليس برنامج تحصيل منفصلًا، بل يغطي دورة العمل كاملة:

**مصدر البيانات ← الاستلام ← الفحص ← المعاينة ← الاعتماد ← الترحيل ← الخطة ← الزيارة ← التحصيل أو البيع ← تسليم النقدية ← المطابقة ← الإغلاق اليومي ← تقرير الإدارة**

يعمل النظام عبر:

- الهاتف.
- الكمبيوتر.
- المتصفح.
- واجهة عربية RTL كاملة.
- تصميم متجاوب.
- PWA قابلة للتثبيت.
- وضع مناسب للاتصال الضعيف.
- مزامنة آمنة للعمليات المؤجلة.

---

## 4. المبادئ الحاكمة غير القابلة للتجاوز

1. فصل SR عن RG في جميع الحركات والحسابات والتقارير.
2. لا يُخزَّن رصيد العميل كرقم قابل للتعديل مباشرة؛ الرصيد نتيجة دفتر حركات.
3. لا حذف لحركة مالية معتمدة؛ التصحيح بحركة عكسية موثقة.
4. لا ترحيل لبيانات حساسة دون معاينة واعتماد.
5. لا رقم دون مصدر وتاريخ قطع بيانات وإصدار معادلة.
6. لا خطة تشغيلية من بيانات غير معتمدة.
7. لا وقود دون خطة وهدف ونتيجة.
8. لا تعتبر الزيارة ناجحة دون نتيجة موثقة.
9. لا ينقل العميل من مندوب إلى آخر دون قرار موثق.
10. لا تعتبر المطابقة نهائية قبل اعتماد الفرق وترحيله كحركة رسمية.
11. لا يسقط الدين بسبب الإفلاس أو الإغلاق أو التوقف.
12. لا يعتمد منشئ الحركة المالية الحساسة عمليته بنفسه.
13. لا تحفظ الأسرار أو بيانات العملاء داخل GitHub.
14. كل تعديل أو اعتماد أو تصدير حساس يسجل في سجل التدقيق.
15. الوقت الرسمي وقت الخادم، وليس ساعة جهاز المستخدم.
16. كل طلب مالي أو تشغيلي من الهاتف يحمل `idempotency_key` لمنع التكرار.

# القسم الأول: المستخدمون والأدوار والصلاحيات

## 5. الأدوار الأساسية

### 5.1 المالك أو المراقب العام

- قراءة جميع بيانات فرع عدن.
- مشاهدة المؤشرات والتقارير وسجل التدقيق.
- لا يحصل تلقائيًا على صلاحية تعديل الحركات.

### 5.2 مدير فرع عدن

- اعتماد الخطط.
- اعتماد الاستثناءات.
- قرارات منع أو إعادة السماح بالآجل.
- اعتماد المطابقات.
- مراجعة الوقود.
- اعتماد الإغلاق اليومي.
- معالجة الحالات الحرجة.

### 5.3 الحسابات والصندوق

- مراجعة السندات.
- استلام النقدية.
- إنشاء دفعات التسليم.
- مطابقة الصندوق.
- ربط التحصيل بالفواتير.
- إدارة الفروقات وفق الصلاحية.

### 5.4 أمين المخزن

- الأصناف والمخزون.
- الحركات الداخلة والخارجة.
- العهد والتحميل للمندوبين.
- المرتجعات والتالف والجرد.

### 5.5 المندوب

- مشاهدة عملائه وخطته فقط.
- تسجيل الزيارة والنتيجة.
- تسجيل التحصيل والوعد والطلب.
- رفع الإثباتات.
- متابعة العمليات المرتجعة إليه.

### 5.6 المدقق أو المراجع

- قراءة البيانات.
- قراءة سجل التدقيق.
- مراجعة التقارير.
- لا يعدل العمليات التشغيلية.

### 5.7 مدير النظام

- إدارة المستخدمين والأدوار والإعدادات التقنية.
- لا يملك تلقائيًا اعتماد الحركات المالية.
- لا يستطيع تعديل سجل التدقيق من الواجهة.

## 6. مستويات الصلاحية

تُبنى الصلاحيات على أربعة مستويات:

1. **صلاحية العملية:** مشاهدة، إنشاء، تعديل، إرسال، مراجعة، اعتماد، عكس، تصدير.
2. **صلاحية السجل:** سجلات المستخدم، عملاؤه، منطقته، فريقه أو جميع السجلات.
3. **صلاحية الحقل:** إخفاء أو إظهار أو منع تعديل حقول حساسة.
4. **صلاحية المرحلة:** ما يُسمح به حسب حالة العملية.

يجب إنشاء مصفوفة صلاحيات مستقلة قبل إطلاق كل وحدة.

# القسم الثالث: النواة المالية ودفتر الحركات

## 9. القاعدة المحاسبية

الرصيد لا يُعدل مباشرة.

الرصيد هو ناتج تجميع دفتر حركات غير قابل للمسح، مع حركات عكسية عند التصحيح.

الجداول الأساسية المقترحة:

- `customer_ledger_entries`
- `invoices`
- `invoice_lines`
- `invoice_allocations`
- `collections`
- `payment_allocations`
- `cash_movements`
- `cash_handover_batches`
- `reversal_entries`
- `opening_balances`
- `credit_notes`

## 10. قواعد التحصيل والتوزيع

يجب توثيق وتنفيذ ما يلي:

- التحصيل المخصص لفاتورة بعينها.
- التحصيل غير المخصص.
- توزيع التحصيل على أقدم فاتورة وفق سياسة معتمدة.
- التحصيل الجزئي.
- السداد الزائد.
- المرتجع والخصم والتالف.
- الرصيد الافتتاحي.
- عكس التحصيل الخاطئ.
- الفرق بين رصيد النظام، رصيد العميل، ورصيد المطابقة.

لا يؤثر التحصيل على الرصيد أو الصندوق إلا عند بلوغ المرحلة المحددة في State Machine.

## 11. تعريف العملات

يُنشأ جدول عملات ثابت، ولا تستخدم نصوص حرة.

لكل عملة:

- كود داخلي.
- اسم كامل.
- رمز عرض.
- عدد المنازل العشرية.
- قواعد التقريب.
- سلسلة ترقيم مستقلة عند الحاجة.

قواعد إلزامية:

- لا تجمع فاتورة واحدة عملتين.
- لا تجمع حركة مالية واحدة عملتين.
- لكل عميل حساب مستقل لكل عملة.
- لا يوجد تحويل تلقائي بين SR وRG.
- أي تحويل استثنائي يحتاج سياسة واعتمادًا وحركة مستقلة.

# القسم السابع: التحصيل والصندوق وسلسلة العهدة

## 17. دورة التحصيل

**مسودة ← مقدم للمراجعة ← مراجع ← مستلم بالصندوق ← مطابق ← مغلق**

مسارات إضافية:

- مرفوض.
- معاد للمندوب.
- متعارض.
- يحتاج قرار مدير.
- ملغى بحركة عكسية.

بيانات التحصيل:

- العميل والمندوب.
- التاريخ والوقت من الخادم.
- العملة والمبلغ.
- طريقة الدفع.
- السند أو التحويل.
- الإثبات.
- حالة التسليم.
- دفعة التسليم.
- مستلم النقدية.
- حالة الاعتماد.
- الفارق.
- `idempotency_key`.

## 18. سلسلة عهدة النقدية

يجب دعم:

- عهدة المندوب.
- وردية الصندوق.
- دفعة تسليم نقدية.
- محضر استلام.
- تسليم جزئي.
- فرق زيادة أو نقص.
- إيداع بنكي.
- إغلاق الصندوق حسب العملة.

يجب أن يكون معلومًا دائمًا:

- من استلم من العميل؟
- من حمل النقدية؟
- متى سلمها؟
- لمن سلمها؟
- هل استلمها الصندوق؟
- هل أودعت؟
- من أغلق الفرق؟

## 19. منع التكرار

يمنع النظام:

- تكرار رقم السند.
- تكرار `idempotency_key`.
- إعادة إرسال نفس العملية من الهاتف.
- نفس العميل والمبلغ والتاريخ دون تحذير.
- استيراد السجل نفسه مرتين.

# القسم الحادي عشر: رفع الملفات والاستيراد والبوابات

## 24. أنواع الملفات

- PDF.
- Excel.
- CSV.
- الصور.
- JSON.
- النصوص.

## 25. دورة الملف

**رفع ← تخزين آمن ← SHA-256 ← كشف التكرار ← تصنيف ← معاينة ← استخراج خام ← فحص ← اعتماد ← ترحيل**

الأنواع التشغيلية:

- أعمار ديون.
- تحصيلات.
- مبيعات.
- عملاء.
- وعود.
- زيارات.
- وقود.
- مخزون.
- مطابقات.
- عملاء مفلسون.
- مصادر عامة.

الصفوف الناقصة أو المتعارضة تدخل منطقة حجر ومراجعة، ولا ترحل مباشرة.

## 26. بوابات الاعتماد

### Gate 1 — تسجيل المصدر

الملف، النوع، المالك، التاريخ، المندوب والوجهة.

### Gate 2 — فحص البيانات

النواقص، العملات، العملاء، التكرار، التعارض والصفوف التجريبية.

### Gate 3 — معاينة الترحيل

عرض ما سيضاف وما سيرفض دون كتابة تشغيلية.

### Gate 4 — التنبيهات والقرارات

استخراج الحالات الخطرة والوعود والفوارق والمهام.

### Gate 5 — الخطط

بناء خطط المندوبين من البيانات المعتمدة فقط.

لا توجد عملية Commit خفية.

# القسم الثاني عشر: جودة البيانات وهوية العميل

## 27. هوية العميل ومنع التكرار

يستخدم النظام `customer_id` داخليًا دائمًا، مع جدول معرفات خارجية لكل مصدر.

يجب دعم:

- المطابقة بالرقم والهاتف والاسم والموقع.
- كشف التشابه دون دمج تلقائي.
- دمج العملاء بموافقة معتمدة.
- حفظ تاريخ السجلات المدمجة.
- منع الدمج بسبب تشابه الاسم فقط.
- دعم فروع العميل أو مواقعه المتعددة دون إنشاء عميل مكرر.

## 28. فحوص الجودة

- عميل بلا رقم.
- رقم مكرر.
- اسم مكرر برقم مختلف.
- عميل بلا مندوب.
- عميل بلا هاتف أو موقع.
- مبلغ بلا عملة.
- خلط SR وRG.
- سند أو فاتورة مكررة.
- وعد بلا تاريخ.
- زيارة بلا نتيجة.
- تحصيل بلا سند.
- وقود بلا خطة.
- فرق غير مفسر.
- ملف قديم أو غير معتمد.
- صف تجريبي.
- سجل مرحل سابقًا.

تظهر درجة جودة عامة ولكل مندوب وملف، مع المسؤول والموعد المطلوب للإغلاق.

# القسم الخامس عشر: البنية التقنية

## 33. الأسلوب المعماري

يعتمد النظام Modular Monolith منظمًا، مع فصل واضح بين:

- واجهة المستخدم.
- API.
- منطق الأعمال.
- الوصول إلى البيانات.
- معالجة الملفات.
- المهام الخلفية.
- التقارير.
- الإشعارات.

المكونات المستهدفة:

- TypeScript.
- واجهة Web/PWA عربية RTL.
- PostgreSQL كنواة تشغيلية رسمية.
- Worker للعمليات الطويلة.
- Queue للمهام.
- Scheduler للمهام الدورية.
- تخزين آمن للمرفقات خلف واجهة موحدة.
- CI/CD عبر GitHub Actions.

الاستضافة يجب أن تبقى قابلة للنقل. لا ترتبط بنية النظام بمزود واحد. يمكن اعتماد Cloudflare Workers مبدئيًا مع PostgreSQL خارجي، شريطة اجتياز اختبارات التوافق والأداء والمهام الخلفية.

## 34. العمليات الطويلة

لا تنفذ داخل طلب متصفح عادي:

- معالجة Excel وPDF.
- OCR.
- تقارير كبيرة.
- استيراد آلاف الصفوف.
- إنشاء الخطط.
- النسخ الاحتياطية.
- الإشعارات الجماعية.

تستخدم Queue مع حالة وظيفة، إعادة محاولة، سجل أخطاء وDead-Letter handling.

## 56. الحكم المهني النهائي

المشروع لا يحتاج مزايا إضافية قبل تثبيت الأساس.

الأولويات هي:

1. دفتر مالي موثوق يولد الأرصدة.
2. نموذج بيانات وعلاقات نهائي.
3. حالات واعتمادات وصلاحيات دقيقة.
4. خطة نقل وربط البيانات الحالية.
5. معادلات موثقة للخطط والمخاطر.
6. بنية للملفات والمهام الخلفية.
7. معايير قبول واختبارات.
8. خطة استعادة وتشغيل عند الأعطال.

هذه الوثيقة تحول المتطلبات التشغيلية إلى مرجع بناء ملزم وقابل للتنفيذ، مع حصر النطاق في فرع عدن فقط ودون تعقيد تعدد الفروع.
